# Internationalization

Pylar ships a typed translation layer backed by JSON catalogues and an ambient locale context variable. Translations load at boot, and the `Translator` resolves keys against the active locale with automatic fallback.

## Directory layout

Place JSON files under `resources/lang/<locale>/`:

```
resources/lang/
    en/
        messages.json
        validation.json
    fr/
        messages.json
        validation.json
```

Each JSON file is a flat or nested object. Keys are prefixed by the filename stem and joined with dots:

```json title="resources/lang/en/messages.json"
{
    "welcome": "Welcome, {name}!",
    "goodbye": "See you later."
}
```

This produces the translation keys `messages.welcome` and `messages.goodbye`. Nested objects are flattened recursively:

```json title="resources/lang/en/validation.json"
{
    "required": "The {field} field is required.",
    "email": { "invalid": "Not a valid email address." }
}
```

This yields `validation.required` and `validation.email.invalid`.

## Configuration

```python title="config/i18n.py"
from pylar.i18n import I18nConfig

config = I18nConfig(default="en", fallback="en")
```

## Using the Translator

Resolve `Translator` from the container and call `get()`:

```python
from pylar.i18n import Translator

translator = container.make(Translator)

translator.get("messages.welcome", placeholders={"name": "Alice"})
# => "Welcome, Alice!"

translator.get("messages.welcome", locale="fr", placeholders={"name": "Alice"})
# => "Bienvenue, Alice !"
```

Missing keys return the key string itself rather than raising, so untranslated text is visible without crashing the request.

## Pluralization

Use `choice()` with CLDR-style suffixed keys (`.zero`, `.one`, `.few`, `.many`, `.other`):

```json title="resources/lang/en/messages.json"
{
    "items.zero": "No items",
    "items.one": "{count} item",
    "items.other": "{count} items"
}
```

```python
translator.choice("messages.items", count=0)   # => "No items"
translator.choice("messages.items", count=1)   # => "1 item"
translator.choice("messages.items", count=42)  # => "42 items"
```

The `count` variable is automatically injected into placeholders. Rules are built in for English, French, German, Spanish, Russian, Ukrainian, Belarusian, and Polish.

## Ambient locale

The `with_locale` context manager sets the active locale for a block of code. The `Translator` reads it automatically so you do not need to pass `locale=` on every call:

```python
from pylar.i18n import with_locale, current_locale

with with_locale("fr"):
    print(current_locale())          # => "fr"
    print(translator.get("messages.welcome", placeholders={"name": "Alice"}))
    # => "Bienvenue, Alice !"
```

## AcceptLanguageMiddleware

Register `AcceptLanguageMiddleware` in your HTTP kernel to set the locale from the request's `Accept-Language` header automatically:

```python
from pylar.i18n import AcceptLanguageMiddleware

class HttpKernel:
    middleware = [
        AcceptLanguageMiddleware,
        # ...
    ]
```

The middleware parses quality factors, picks the highest-priority locale that has a loaded catalogue, and installs it via `with_locale` for the duration of the request. If no match is found it falls back to the configured default.

## Usage in Jinja templates

The `Translator` can be exposed as a Jinja global in your view provider's `boot()`:

```python
renderer.environment.globals["__"] = translator.get
```

Then in templates:

```html+jinja
<h1>{{ __("messages.welcome", placeholders={"name": user.name}) }}</h1>
```
